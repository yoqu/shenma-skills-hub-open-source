# SkillStack 项目风险分析

日期：2026-05-27

范围：基于当前仓库静态代码、路由、Controller、Flyway migration、测试文件、项目文档与本地启动观察进行分析。本文只记录风险点，不代表线上验收结论。

## 总体判断

项目主体功能已经较完整，团队内 Skill / Prompt / Suite 的创建、审核、管理、发现、安装、通知、权限与 CLI 消费均已有代码实现。

当前主要风险不在“是否有页面或接口”，而在生产化闭环、环境一致性、文档同步、占位入口收口、真实外部服务接入和全量回归验证。

## 风险清单

| 编号 | 等级 | 风险点 | 证据 | 影响 | 建议动作 |
| --- | --- | --- | --- | --- | --- |
| R-001 | 高 | 运行文档与实际启动方式不一致 | `AGENT.md` / `plan.md` 仍描述一键脚本启动 MySQL、`docker compose`、本地 `3307`；当前已改为后端使用 `application-dev.yaml` 的环境数据库，脚本不再管理 MySQL | 新人或自动化按文档启动会误操作，甚至误判环境故障 | 同步更新 `AGENT.md`、`backend/README.md`、`plan.md` 中本地运行和数据库说明 |
| R-002 | 高 | dev 环境数据库与 migration 版本存在漂移风险 | 启动日志显示远端 `skill_dev`；曾出现 schema version 27 与本地 migration 认知不一致的告警 | 不同分支、不同环境连接同一 dev 库时，Flyway 可能出现 future migration / schema drift | 明确 dev 库归属；建立 migration 发布规则；禁止多人随意对共享 dev 库跑未合并 migration |
| R-003 | 高 | 短信验证码仍是 mock / 内存实现 | `AuthService` 注释标明短信验证码使用内存 Map，生产应替换短信服务 + Redis | 注册、登录、换绑手机号无法满足生产可靠性、安全性和分布式部署 | 接入真实短信网关；验证码存储迁移 Redis；补充限流、防刷与审计 |
| R-004 | 高 | OAuth 依赖环境配置，默认未配置时不可用 | 后端启动日志出现 `app.oauth.state-secret is not configured; OAuth endpoints will return 503` | OAuth 登录按钮存在但运行环境不完整时用户无法登录 | 明确 `.env` 必填项；启动时给出更清晰健康检查；前端根据 provider 可用性隐藏不可用入口 |
| R-005 | 中 | 通知偏好已持久化，但外部投递闭环未完成 | 代码有站内通知和偏好；计划文档中邮件/投递器仍列为延伸项 | 用户开启偏好后可能误以为邮件或外部渠道已经生效 | 将通知类型分为“已接入站内”和“未接入外部投递”；补充投递 worker 设计 |
| R-006 | 中 | 部分 UI 仍可能是占位或弱闭环 | 历史测试文档记录关注/私信、危险操作、部分 Dashboard 审计展示等占位风险；当前仍需逐页确认 | 用户可能点击无效入口，测试也可能误判为功能完成 | 建立占位入口清单：禁用、隐藏或明确标注“暂未开放”；补 Playwright smoke |
| R-007 | 中 | 前端仍大量依赖 `@/mocks` 类型和 adapter | `frontend/src/api/data.ts`、`team/admin/_shared/api.ts` 中大量把后端 DTO 映射到 mock 类型 | 类型边界不稳定，后端字段变化时容易静默降级为空值或默认值 | 逐步建立正式 API DTO 类型；减少 mock 类型在真实页面中的复用 |
| R-008 | 中 | 文档中的完成状态可能过度乐观 | `plan.md` 标记 Phase 0-3 全部完成，但后续新增了 prompt、OAuth、平台超管等超出原计划的模块 | “已完成”口径无法反映真实验收范围和剩余工作 | 重写一份当前版本能力矩阵，按模块标注：已实现、可用但需配置、占位、未实现 |
| R-009 | 中 | PAT 权限范围需要持续审计 | `JwtAuthFilter` 对 `lst_` token 做路径白名单；当前覆盖 skill / prompt / suite 消费路径 | 新增 API 时可能遗漏 PAT 权限边界，导致过宽或过窄 | 给 PAT 白名单补集中测试矩阵；新增消费接口时必须同步权限用例 |
| R-010 | 中 | 平台超管能力涉及高危操作，需确认审计闭环 | 超管支持用户启停、重置密码、团队启停、成员管理、OAuth provider、站点配置 | 缺少审计或回滚能力会造成运维风险 | 确认 admin audit 是否真实写入并可查询；对高危操作加二次确认和审计查询 |
| R-011 | 中 | 自动化测试覆盖后端较多，前端 E2E 覆盖不足 | 后端测试文件较完整；前端主要看到单测和历史测试文档，缺少当前全量 E2E 证据 | 页面交互、权限跳转、真实 API 联调风险不易被 CI 捕获 | 建立最小 Playwright 回归：登录、团队工作台、创建 skill、审核、prompt、suite、超管 |
| R-012 | 低 | 启动脚本与文档仍需收口 | `scripts/services.sh` 已改为只管理前后端、自动安装前端依赖、用端口 LISTEN 判断；文档未同步 | 后续维护者可能恢复旧 MySQL/Docker 假设 | 将脚本行为写入 README；说明 stop 只停前后端，不停数据库 |

## 待确认风险

以下风险需要结合真实业务需求或运行环境确认，不能仅凭代码判断直接认定为缺陷：

- 共享 `skill_dev` 数据库是否允许多人共用，以及 migration 版本 27 是否来自已合并代码。
- OAuth 在当前阶段是否必须可用，还是只作为可选登录方式。
- 关注、私信、外部通知投递是否属于当前版本范围。
- 平台超管审计是否要求 UI 可查询，还是仅保留数据库审计即可。
- 前端保留 mock 类型是否只是类型复用过渡，还是需要纳入本期清理范围。

## 建议优先级

1. 先同步运行文档与启动脚本行为，避免环境误用。
2. 明确 dev 数据库和 Flyway migration 管理规则。
3. 补齐 OAuth / 短信 / 通知投递的生产配置与降级策略。
4. 建立当前版本能力矩阵，清理或禁用占位入口。
5. 补最小 E2E 回归，覆盖核心团队工作流和平台超管高危操作。
