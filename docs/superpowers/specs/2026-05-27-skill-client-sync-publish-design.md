# Skill 客户端、同步与持续发布平台 设计

- 日期：2026-05-27
- 状态：产品需求层已对齐，待实现选型与技术 spec 二次细化
- 影响范围：新增桌面客户端 / 后端订阅、统计、推荐包、下架能力 / 前端"我的订阅 / 我的发布"页 / CLI 与客户端共享同步语义
- 配套关系：本文档只锁定**产品需求**，后续会拆出实现 spec（客户端形态、同步引擎、API、数据模型）

## 1. 背景与问题

当前 SkillStack 完成了 skill 提交、审核、版本管理、CLI 安装与下载，但团队 skill 协作仍存在几条断层：

1. **使用者** 装完 skill 之后版本一直停在过去，作者更新了无人知晓；多台设备工具箱无法跟着账号走。
2. **作者** 每次发新版要重新走 zip → upload，迭代摩擦大；做完之后看不到有谁在用、用得怎么样，反馈闭环缺失。
3. **团队管理者** 看不见团队 skill 资产的健康度，不知道哪些被广泛使用、哪些已经沉睡，也没有"团队默认推荐 skill"的机制让新人快速对齐。

平台现有 web + CLI 形态对这些痛点都只能局部缓解，需要补一层**桌面客户端 + 订阅 / 通知 / 统计 / 推荐能力**形成完整闭环。

## 2. 设计目标

1. 三类用户角色的真实工作流要跑通：使用者、作者、团队管理者。
2. 产品语义清晰：**订阅**与**安装**是两个独立状态，"我的 skill 集"跟着账号走且跨团队跨广场。
3. 同步与升级有可控的批量能力；用户多端登录看到一致的订阅，但安装动作仍由每台设备显式触发。
4. 作者发布更新的摩擦尽量低，但治理权下放团队（每个团队自定义审核策略）。
5. 数据可见性默认收紧，作者看聚合数据；用户可显式 opt-in 让作者看到自己；管理者只看团队 skill 的整体指标，不窥探成员个人 skill 清单。
6. 通知触发集精挑细选，避免噪音爆炸。

## 3. 非目标

以下能力**不在本期范围内**，避免范围漂移：

- 角色 4：本地修改 skill 后反哺给原作者（本地分叉 / merge 回上游）
- 使用者手动版本回滚（装了 v2 切回 v1）
- "我的 skill 配置"导入导出（通过 suite 承担"分享一组 skill"的诉求）
- 试用 / 预览模式（短期试装过期自动卸载）
- skill 类型分化（纯 prompt vs 含可执行脚本统一对待）
- 多作者协同维护单个 skill（v1 单 owner 模型）
- 复杂审核分流（按 diff 内容判断是否敏感字段、敏感度评分）
- 自动行为推荐（"团队 80% 都装了 X"这种基于行为的推荐 UI）
- 跨平台桌面推送 / 邮件 / IM 等通知通道实现细节（留到实现 spec 决定）
- 订阅作者本身（关注作者的新发布）

## 4. 服务的用户角色

| 角色 | 工作中关心什么 | 本期被服务的具体诉求 |
|------|--------------|-----------------|
| 使用者 | 我有哪些 skill 可用、是不是最新、跨设备一致 | J1 |
| 作者 | 我的 skill 怎么持续发布、有多少人在用 | J2、J3 |
| 团队管理者 | 团队资产健康度、新人入职能否快速对齐 | J4、H1（团队推荐包） |

明确排除（v1 不做）：

- 角色 4：在本地分叉 skill 并回推给原作者

## 5. 核心用户旅程

### J1 · 使用者保持团队 skill 同步与新鲜

**入口**：客户端启动 / 站内打开"我的 skill"。

**关键步骤**：

1. 新人加入团队 → 看到团队管理者维护的"团队推荐包"（H1，由一个团队级公开 suite 承载） → 一键安装清单内全部 skill。
2. 用户可在广场 / 团队库 / 用户个人页订阅任意自己有权限看到的 skill（跨团队、跨广场）。
3. 客户端 / 站内"我的 skill"按 skill 显示：订阅状态、本机是否安装、本机版本、远端最新版本、是否有可升级、是否被下架。
4. 远端任一已订阅 skill 出新版 → 通知（N1）；已安装的 skill 出新版 → 强通知（N2）。
5. 用户可单个升级、跳过此版本（用户在此 skill 上保持当前版本，直到下一次新版本到来才再次提醒）、或一键批量升级所有"已安装 + 有新版"的 skill。
6. 卸载只影响本机；退订是用户全局动作，所有设备同步从订阅列表移除（但本机已装副本不强制删除，由用户决定）。
7. 新设备首次启动看到完整订阅列表，每个 skill 标"未安装"，并提供"一键同步全部已订阅 skill 到本机"按钮。

### J2 · 作者从本地持续发布

**入口**：客户端中作者的本地 skill 工作区 / 站内"我的发布"页。

**关键步骤**：

1. 作者识别一个本地目录为某个已发布 skill 的本地副本（首次发布时建立绑定关系）。
2. 作者在本地反复迭代，准备好后点"发布新版本"。
3. 客户端把本地内容打包，按团队 `review_policy` 配置决定走审核队列还是直接发布。
4. 审核通过（或免审）→ 版本立即对所有订阅者可见，N1 / N2 通知触发。
5. 若审核被拒：作者在"我的提交"页看到拒绝原因，手动修改后重新提交（同一版本号或递增由系统按现有 skill-version 规则处理）。
6. 已发布的版本**不可撤回**；作者若想停止维护，只能对整个 skill 执行**下架**（详见第 9 节）。

### J3 · 作者看到自己 skill 的使用反馈

**入口**：站内"我的发布" / 单个 skill 的作者视角看板。

**展示指标**（按隐私模型 P3 限定为聚合）：

1. 订阅数（多少人订阅了我）
2. 安装数（多少台设备装了我）
3. 活跃数（最近 N 天有使用行为的用户数，N 待定，建议 7 / 30 / 90）
4. 版本分布（每个版本当前还有多少装机量，多少人停在旧版）
5. 时间趋势：7 / 30 天的安装与升级曲线
6. opt-in 用户名单：仅展示主动选择"对作者公开身份"的用户（用于建联早期用户 / 后续 v2 反馈循环）

### J4 · 团队管理者看团队 skill 资产健康度

**入口**：团队 admin 后台 → 资产看板。

**展示指标**（按隐私模型 P3 限定为团队层聚合）：

1. 团队 skill 总数 / 活跃 vs 沉睡（90 天无新安装定义为沉睡，可调）
2. 团队 Top 使用榜
3. 每个 skill 在本团队成员中的覆盖率（订阅率、安装率）
4. 每个 skill 的版本一致性：团队成员都在哪个版本上，旧版本残留情况
5. **不显示**："某成员装了哪些 skill"这种个人维度的视图

## 6. 核心产品规则

### 6.1 订阅 ⊥ 安装

订阅与安装是两个独立状态。可能组合：

| 订阅 | 安装 | 状态含义 |
|------|------|--------|
| ✓ | ✓ | 常规使用中，跟随升级 |
| ✓ | ✗ | 仅关注，未本机安装；典型场景：跨设备订阅未在本机落地、纯观察 |
| ✗ | ✓ | 装了但锁版本，不跟随升级；典型场景：使用者主动停止订阅以避免变更 |
| ✗ | ✗ | 无关 |

订阅是用户级（云端唯一），安装是设备级。

### 6.2 订阅范围

用户可以订阅任何自己有权限看到的 skill：

- 自己所在团队的所有 skill
- 公共广场（PUBLIC + APPROVED）上的所有 skill
- 通过邀请加入的其他团队的 skill

订阅集合跨团队、跨广场，构成"我的 skill 工具箱"，跟随用户账号而非团队。

### 6.3 多端语义

- **订阅状态**：云端唯一，多端实时同步。
- **安装状态**：设备级独立。新设备登录看完整订阅列表 + 一键同步选项。
- **卸载**：只影响本机，不解除订阅。
- **退订**：用户全局动作，所有设备同步移除订阅；已安装本机副本保留，由用户决定是否本地卸载。

### 6.4 H1 · 团队推荐包 / 落地为团队公开 suite

- 由团队管理者显式维护，是一个**团队级公开 suite**。
- 新人首次进入团队工作区时主动展示。
- 任意时刻成员可在团队库找到并一键安装。
- 套件是一次性安装清单：装完后清单内的每个 skill 各自独立进入用户的订阅列表，后续 N1 / N2 / N4 按 skill 维度通知。

### 6.5 套件（suite）的定位

- 套件是**一次性 install 清单**，不支持订阅 suite 本身。
- 套件用于"管理者维护推荐包"和"作者 / 用户分享一组 skill 给同事"两个场景。
- 套件内容后续变化不会主动推送给装过它的用户（订阅在 skill 维度，不在 suite 维度）。

### 6.6 依赖关系

- 创建 skill 时可声明依赖其它 skill。
- 用户订阅 / 安装一个 skill 时，自动联动其依赖（订阅链上的全部 skill）。
- 系统必须**拒绝循环依赖**（创建 / 更新 skill 时做闭环检测，发现成环立即拒绝）。
- v1 不支持版本范围（"A 依赖 B >= 1.2"），简化为"A 依赖 B"，B 跟随最新。
- 依赖被下架时：A 仍可继续发布、继续被订阅；平台提示作者"你声明的依赖 B 已下架，建议解除依赖"；已装 A 的用户本机不受影响。
- 退订 / 卸载不连带：用户退订 A 时不自动退订 B；卸载 A 不连带卸载 B。

### 6.7 审核策略

- 不在产品层硬编码"按变更类型分流"等复杂规则。
- 每个团队在 settings 配置 `review_policy`，v1 支持：
  - `always`：每个版本都进审核队列
  - `first_only`：首版过审，作者后续修改自己 skill 免审
  - `none`：完全免审，直接发布
- Journey 2 的发布流读这个配置决定走向。
- 后台保留所有发布事件审计日志，管理员可对任意已发布版本追加 review。

### 6.8 版本不可撤回、skill 可下架

- 任意一个**已发布并通过当时审核规则的版本**是不可变历史，作者不能 withdraw / rollback。
- 作者要"停止维护"只能对整个 skill 执行**下架**：
  - 谁能下架：作者本人；本团队 skill 由团队管理者也可强制下架（治理需要）。
  - 下架后：搜索 / 广场 / 团队库立即移除，不可再被新订阅。
  - 已订阅用户：订阅保留但标 `(已下架)`，不再产生 N1；触发 N8 通知一次。
  - 已安装用户：本机副本完全保留，可继续运行，UI 上挂"已下架"灰色 badge。
  - 是否可重新上架：可，下架是软状态，作者 / 管理者可恢复 `active`。
  - 下架期间不能继续发新版本，需先恢复上架。
  - 所有历史版本永久归档可读。

### 6.9 拒绝后的提交反馈

- 审核拒绝原因在"我的提交"页对作者可见。
- 作者修改后手动重新提交，沿用现有 skill-version 提交规则。
- 不做自动重试 / 二次自动转人工等复杂分流。

## 7. 隐私模型（P3）

| 数据 | 作者本人 | 团队管理者（本团队 skill） | 平台 / 其他用户 |
|------|--------|----------------------|--------------|
| 自己 skill 的订阅 / 安装 / 活跃聚合数 | ✓ | ✓ | 公共 skill 可见聚合 |
| 版本分布、时间趋势 | ✓ | ✓ | ✗ |
| 具体哪些用户订阅 / 安装了我的 skill | 仅 opt-in 用户 | ✗ | ✗ |
| 某个用户的全部 skill 清单 | ✗ | ✗ | ✗ |
| 团队内某 skill 的覆盖率 | — | ✓ | ✗ |

后台数据库存全量明细供聚合查询、审计和风控，但 UI 严格按上表暴露。

## 8. 通知触发集（v1）

| 编号 | 事件 | 强弱 | 备注 |
|------|------|-----|------|
| N1 | 我订阅的某个 skill 出新版本 | 弱（可聚合 digest） | Journey 1 主路径 |
| N2 | 我已安装的 skill 出新版本 | 强 | 比 N1 优先级高 |
| N4 | 团队管理者把某 skill 加入团队推荐包 | 弱 | H1 落地路径。N4 是团队成员级事件——只要管理者把某 skill 加入团队推荐 suite，所有团队成员都收到通知（与他们是否订阅 suite 无关，因为 suite 本身不可订阅） |
| N5 | 我提交的 skill 通过 / 拒绝审核 | 强 | 个人事件 |
| N8 | 我已订阅或已安装的 skill 被下架 | 强 | 让用户知道工具集发生变更 |

不进 v1：

- N3 订阅但未装的 skill 新版（噪音）
- N6 关注作者的新 skill 发布（依赖"订阅作者"模型）
- N7 我的 skill 被某人订阅 / 安装（违反 P3）
- N9 本地 skill 因远端 force-update 被覆盖前的预警（角色 4 才需要）

通知通道（站内 / 桌面推送 / 邮件等）由后续实现 spec 决定，本文档只锁定**事件**。

## 9. 数据可见性与角色权限要点（产品口径）

- 公共广场始终是匿名可访问，仅展示 PUBLIC + APPROVED skill。
- 团队库需团队成员身份。
- 跨团队订阅：用户对 B 团队 skill 的访问以"该 skill 是否对该用户可见"判定；不可见即不可订阅。
- 团队管理者对本团队 skill 的统计、下架、推荐包配置具备完整权限；对成员个人订阅 / 安装清单无访问权。
- 作者对自己 skill 的统计完整可见，但用户身份层默认聚合，仅 opt-in 用户可见名单。

## 10. v1 范围划界总结

**v1 必须做**：

- 桌面客户端（形态在实现 spec 决定，仍可与现有 CLI / web 并存）
- 同步引擎：本地 ↔ 远端订阅 / 安装 / 版本状态同步
- 订阅系统（用户级、跨团队、跨广场）
- 安装系统（设备级、批量升级、跳过版本）
- 通知触发集 N1 / N2 / N4 / N5 / N8
- 作者发布流（本地副本 → 一键发布 → 走团队审核策略 → 拒绝反馈）
- 作者统计看板（聚合 + opt-in 名单）
- 团队管理者看板（团队资产健康度）
- 团队推荐包（团队级公开 suite）
- 团队审核策略配置（`always` / `first_only` / `none`）
- skill 下架能力 + 已发布版本不可撤回
- skill 依赖声明 + 循环依赖检测 + 订阅 / 安装联动

**v1 明确不做**：见第 3 节。

## 11. 后续待解的设计点（留给实现 spec / writing-plans）

仅产品层不再展开，写实现 spec 时必须解决：

1. **客户端形态**：Electron / Tauri / VSCode 扩展 / Web + 本地 daemon 的 trade-off 与选型。
2. **同步引擎抽象**：本地 manifest / lockfile 设计、增量更新算法、跨平台路径处理。
3. **CLI 与客户端的关系**：共享同一同步内核还是各自实现；CLI 是否要支持订阅。
4. **通知通道**：站内 / 桌面推送 / 邮件如何落地，未读模型与现有 unified notification center 的复用关系。
5. **后端能力补齐**：订阅表、设备表、安装事件表、统计聚合（实时 vs 离线聚合）、推荐 suite 标识、审核策略字段、skill 下架状态机、依赖关系表与闭环检测。
6. **前端补齐**：作者"我的发布"看板、管理者资产看板、推荐包维护界面、跨设备订阅可视化。
7. **失败 / 降级**：客户端离线行为、网络中断后的同步重放、服务端不可用时本地体验。
8. **多设备身份**：同一账号在多端登录的认证、设备清单、登出某台设备语义。
9. **下架的版本与依赖**：依赖了已下架 skill 的 skill 在统计 / UI 上的呈现细节。
10. **活跃用户的口径**：什么样的行为算"使用"（启动 client / 调用 skill / 安装 / 升级），决定 J3 的活跃数怎么算。
11. **管理者强制下架时作者的通知**：当团队管理者强制下架本团队作者的 skill 时，作者本人是否、以何种形式被通知（建议复用 N8 但显式标记 "由管理者下架"）。
